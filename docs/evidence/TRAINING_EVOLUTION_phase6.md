# Training Pipeline Evolution & Validation

> **Note**: This is a historical document from Phase 6 development. Node names have been updated to current logical designations.

## Overview
This document chronicles the successful testing, fixing, and end-to-end validation of the Home AI Lab **Phase 6 Agent Training Runtime**. The goal of Phase 6 was to establish a fully reproducible, automated loop for generating synthetic data, fine-tuning reasoning models via GRPO (Generative Reward Policy Optimization), and converting those models to run directly in the local GPU cluster.

## Pipeline Validation Run (March 22, 2026)

We executed the full training pipeline to validate its stability, hardware utilization, and path persistence.

### 1. Synthetic Data Generation (`synthetic_gen.py`)
- **Target**: 50 conversational traces
- **Outcome**: Successfully triggered the BMO Architect persona to simulate multi-turn interactions with various tools and routing scenarios.
- **Output**: 50 traces successfully recorded in JSONL format.
- **Issue Resolved**: Fixed a critical parameter mismatch (`--target` instead of `--target_count`) and utilized `MSYS_NO_PATHCONV=1` to prevent MinGW trailing path corruption.

### 2. QLoRA GRPO Fine-Tuning (`grpo_trainer.py`)
- **Base Model**: `Qwen/Qwen2.5-Coder-7B-Instruct` (15.2 GB)
- **Dataset**: `synthetic_20260322_153114.jsonl` (50 samples)
- **Configuration**: 4-bit quantization, LoRA rank 16 (alpha 32), 75 total epochs.
- **Hardware Profile**: The run fully occupied the RTX 5060 Ti, maintaining ~12.5GB VRAM usage safely below the 16GB limit, successfully triggering the GPU Mutex system to evict Ollama and ComfyUI for the duration.
- **Metrics**: 
  - **Start**: Reward 0.615 | Loss -0.195
  - **Final**: Reward 1.000 | Loss -0.102
  - **Duration**: ~10.2 hours (36,713 seconds). 
- **Outcome**: Perfectly converged. The LoRA adapter was successfully saved to the output directory.

### 3. GGUF Conversion (`convert_gguf.py`)
- **Execution**: Run via the extraction pipeline, merging the LoRA adapter natively into the model weights using the execution node's CPU RAM to avoid VRAM OOM during serialization.

## Infrastructure & Observability Improvements

During the training pipeline validation, several massive upgrades were made to the core observability stack:

### 1. Fixing Grafana "No Data" via Postgres
**Problem**: The `Training Pipeline` dashboard top stats originally queried Prometheus for metrics tracking total runs and duration. However, because the agent backend was stateless between scripts, Prometheus counters were clearing.
**Solution**: Rewrote all 6 top modules in the Grafana JSON to directly query the persistent `PostgreSQL-Swarm` database.
- Uses `SELECT count(*) FROM swarm.training_runs` for totals.
- Uses `$__timeFilter()` dynamic Postgres buckets for time-series charts.

### 2. cAdvisor GPU Hardware Telemetry
**Problem**: Prometheus is hosted on the Gateway Node, but the heavy GPU containers (`comfyui_gpu`, `ollama_gpu`, `training_runtime`) run exclusively on the Execution Node. Grafana reported "No data" for all Container Memory and CPU usage for the AI stack.
**Solution**: 
- Injected Google `cAdvisor` into `execution_plane/docker-compose.yml` (Execution Node) natively mapped to the Docker daemon.
- Configured Gateway Node Prometheus to hit `192.168.2.101:8080/metrics`.
- **Result**: Complete, real-time hardware tracking for all GPU inference nodes visible instantly in the Hive UI.

### 3. Mount Persistence Overhaul
**Problem**: `training_data` and `training_output` were initially defined as native Docker named volumes (`execution_plane_training_data`), rendering generated JSONL datasets and trained adapters effectively invisible to the Windows host filesystem.
**Solution**: Converted the execution mounts to `./training_data` and `./training_output` bind mounts. Data generated by the AI cluster now immediately populates locally in the open GitHub Workspace for direct user inspection and manual backup.
