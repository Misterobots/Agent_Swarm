# MemPalace Migrations

End-to-end procedures for evolving the MemPalace database schema. Schema is
managed by [Alembic](https://alembic.sqlalchemy.org/) — the runtime app
calls `alembic upgrade head` on boot, so deployed code refuses to start
unless the DB is migrated to a compatible state.

> **Source files:** [`control_plane/mempalace/alembic/`](https://github.com/Misterobots/Agent_Swarm/tree/main/control_plane/mempalace/alembic)

---

## Where the schema lives

| Artifact | Purpose |
|---|---|
| `control_plane/mempalace/alembic.ini` | Alembic config (DB URL is blank — overridden in `env.py` from env vars) |
| `control_plane/mempalace/alembic/env.py` | Async migration runner; restricts to the `mempalace` schema; imports models from `app.database` |
| `control_plane/mempalace/alembic/versions/*.py` | One file per migration |
| `control_plane/mempalace/alembic/script.py.mako` | Template for new migrations |
| `mempalace.alembic_version` (DB) | Single-row table tracking current revision |

The ORM models in [`app/database.py`](https://github.com/Misterobots/Agent_Swarm/blob/main/control_plane/mempalace/app/database.py)
are the source of truth at the Python level — when models and migrations
disagree, autogenerate tells you. Migrations are the source of truth at
the database level — they're what actually runs against Postgres.

## Procedure A — Add a new migration

The everyday workflow when you need to add a column, index, or constraint.

### 1. Edit the model

Make the change in `control_plane/mempalace/app/database.py`. Example:
adding a `tags` column to `Memory`:

```python
class Memory(Base):
    __tablename__ = "memories"
    # ...
    tags = Column(JSONB, server_default=text("'[]'::jsonb"))
```

### 2. Generate the migration

From inside the running container (or any environment with the deps):

```bash
docker compose -f control_plane/docker-compose.yml \
  exec mempalace alembic revision --autogenerate -m "add tags column to memories"
```

This produces `control_plane/mempalace/alembic/versions/000N_add_tags_column_to_memories.py`.

### 3. Review the generated migration

**Always read the autogenerated file before committing.** Common pitfalls:

- pgvector `Vector(768)` columns sometimes render as `sa.NullType()` —
  edit the file to use `from pgvector.sqlalchemy import Vector` and
  `Vector(768)`.
- `server_default` values may need adjustment (autogenerate guesses based
  on the model).
- IVFFlat / HNSW index options aren't always preserved — copy from the
  baseline migration if you're touching vector indexes.

```python
# Example of what to review:
def upgrade() -> None:
    op.add_column(
        "memories",
        sa.Column("tags", JSONB(), server_default=sa.text("'[]'::jsonb")),
        schema="mempalace",
    )

def downgrade() -> None:
    op.drop_column("memories", "tags", schema="mempalace")
```

### 4. Test on a dev DB

Bring up a fresh local instance, let it run all migrations from baseline,
then verify schema matches expectations:

```bash
docker compose -f control_plane/docker-compose.yml down -v mempalace postgres
docker compose -f control_plane/docker-compose.yml up -d
docker compose exec postgres psql -U langfuse -d langfuse -c "\d mempalace.memories"
```

### 5. Commit and deploy

Commit the model change and the generated migration as one unit. Deploy
following [Procedure C](#procedure-c-deploy-a-migration).

## Procedure B — Adopt Alembic on an existing DB

**One-time procedure** for environments that have tables created by the
pre-Alembic `Base.metadata.create_all` and do not yet have an
`alembic_version` row. Skipping the stamp will cause boot to fail with
"relation already exists" when `0001_baseline` tries to recreate tables
that already exist.

### Steps (run on the host where mempalace deploys)

```bash
ssh misterobots@{{ hopper_ip }}
cd ~/Agent_Swarm/control_plane

# 1. Pull the new code (which has alembic baked in)
cd ~/Agent_Swarm && git pull
cd control_plane

# 2. Build the new image (boot will require alembic in PATH)
docker compose build mempalace

# 3. STAMP first — tells alembic the DB is already at baseline state
#    Use `run --rm` so we don't start the long-lived container yet:
docker compose run --rm --entrypoint alembic mempalace stamp 0001_baseline

# 4. Verify the stamp landed
docker compose run --rm --entrypoint alembic mempalace current
# Expected: 0001_baseline

# 5. Now safe to bring up the new container — boot runs `upgrade head`
#    which applies any later migrations (e.g. 0002_add_unique_constraints)
docker compose up -d --force-recreate mempalace

# 6. Confirm
docker compose logs --tail=20 mempalace
docker compose exec mempalace alembic current   # should show the head revision
curl -fsS http://{{ hopper_ip }}:8200/health
```

This procedure was used to introduce Alembic to the live mempalace DB on
2026-05-08. Run it once per environment.

### Pre-deploy backup (recommended)

```bash
docker exec postgres pg_dump -U langfuse --schema=mempalace langfuse \
  > ~/mempalace_backup_$(date +%Y%m%d_%H%M%S).sql
```

The dump is ~1.6 MB per ~150 memories. Keep until the new state is
confirmed stable.

## Procedure C — Deploy a migration

Standard deploy flow once Alembic is already adopted on the target DB.

```bash
ssh misterobots@{{ hopper_ip }}
cd ~/Agent_Swarm
git pull
cd control_plane
docker compose build mempalace
docker compose up -d --force-recreate mempalace
docker compose logs --tail=30 mempalace
```

The boot sequence runs `init_db()` which:

1. `CREATE EXTENSION IF NOT EXISTS vector` (idempotent)
2. `CREATE SCHEMA IF NOT EXISTS mempalace` (idempotent)
3. `alembic upgrade head` (applies any pending migrations)

If migrations fail, the FastAPI app does not start and the container will
fail its `HEALTHCHECK`.

## Procedure D — Roll back a migration

Use sparingly. Migrations are the historical record — preferring "fix
forward" with a new migration is usually safer.

### Roll back one revision

```bash
docker compose -f control_plane/docker-compose.yml \
  exec mempalace alembic downgrade -1
```

The container is still running on the old code, which now mismatches the
DB. Roll the container back to the matching code before the next request:

```bash
git checkout <previous-commit>
docker compose build mempalace
docker compose up -d --force-recreate mempalace
```

### Roll back to a specific revision

```bash
docker compose exec mempalace alembic downgrade 0001_baseline
```

### Restore from backup (last resort)

```bash
docker exec -i postgres psql -U langfuse -d langfuse \
  < ~/mempalace_backup_<timestamp>.sql
```

## Common scenarios

| Scenario | Procedure |
|---|---|
| Adding a column / index / constraint | [A](#procedure-a-add-a-new-migration) (add migration) → [C](#procedure-c-deploy-a-migration) (deploy) |
| Adopting Alembic on a live DB for the first time | **B** (one-time stamp) |
| Routine deploy (no schema changes) | **C** as usual — `upgrade head` is a no-op |
| Bringing up a fresh dev environment | Just `docker compose up`; baseline runs from scratch |
| Migration broke prod, need to revert | **D** (downgrade) + redeploy old image; investigate; fix forward |

## Failure modes

### "relation already exists" at boot

**Cause:** Live DB already has the tables (from pre-Alembic `create_all`)
but no `alembic_version` row exists. Boot tries to run `0001_baseline`
which fails on `CREATE TABLE memories`.

**Fix:** Run [Procedure B](#procedure-b-adopt-alembic-on-an-existing-db).

### "Target database is not up to date"

**Cause:** A schema change exists in the model but no migration was
generated, or a migration was generated but not committed.

**Fix:** Run `alembic revision --autogenerate -m "..."` and review/commit
the result.

### Autogenerate emits `sa.NullType()` for `embedding` columns

**Cause:** Alembic's autogenerator doesn't know about pgvector's `Vector`
type by default.

**Fix:** Hand-edit the migration:

```python
from pgvector.sqlalchemy import Vector

# Replace this:
sa.Column("embedding", sa.NullType())
# With:
sa.Column("embedding", Vector(768))
```

`alembic/env.py` already imports `Vector` so the runtime side is fine —
only the autogen output needs editing.

### "Invalid value for sqlalchemy.url"

**Cause:** `alembic.ini` has `sqlalchemy.url =` (blank) and the env vars
that `env.py` reads (`MEMPALACE_DB_USER` / `_PASSWORD` / `_HOST` / etc.) are
not set in the calling environment.

**Fix:** Run alembic via `docker compose exec mempalace ...` so the
container's env vars (set by compose) apply automatically.

### Live container fails `HEALTHCHECK` after deploy

**Cause:** Migration ran cleanly but the app code references a column or
table that doesn't yet exist (rare — usually means the migration was
incomplete).

**Fix:** `docker compose logs mempalace`, identify the missing schema
element, write a follow-up migration, deploy again.
